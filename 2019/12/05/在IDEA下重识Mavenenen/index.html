在IDEA下重识Maven

关于Maven的基础认识其实早在前期的文章便有过记录，但是随着开发生态和工具的发展，加之前一段时间所在的实习单位所使用的开发工具与原来在校使用的有所出入，所以特此在收悉全新的工具的同时，在回顾一下Maven的相关要点，并在IDEA上进行实践。

还不了解Maven基本要点和安装配置的同学可以去看我的上一篇关于Maven的文章

在IDEA下配置Maven

首先在你的开发环境上安装Maven并配置（Maven从官网上是以压缩包的形式下载的，所以不存在安装问题，只需下载后解压到指定目录即可。）随后安装IDEA，如果是正版用户这里推荐使用JetBrains的Toolbox App安装（如官网所说：“Manage your tools the easy way”）。

关于Maven的配置，这里多说一点。由于我之前在校本地开发的时候直接使用的是Maven安装目录下的config文件中的setting.xml，其中配置的是阿里的Maven仓库地址，但是由于实习公司使用了nexus私服管理依赖，所以在进入公司后重新配置了本地用户路径下的.m2/setting.xml。

在完成上述安装与配置后，我们首先打开IDEA并且创建一个Maven项目。

设置GroupId和ArtifactId（GroupId为项目的组织名，ArtifactId为工程项目名）

随后选择项目保存目录后Next，IDEA便会在指定目录下创建Maven项目

上述的Maven项目为一个空的Maven项目，里面没有任何内容和依赖，这样是为了方便我们在IDAE配置Maven。（有些情况下，我们在选择默认setting.xml配置时，可能会出现在自己本地用户目录下找不到.m2文件，这是因为.m2文件并不是用户能够自己创建的，其是一个隐藏文件，是在创建Maven项目后自动创建的。所以这里选择创建一个新的Maven项目来演示，也是为了方便用户可以在IDEA中配置Maven时可以找到.m2/setting.xml）

接下来我们开始在IDEA中配置Maven。首先打开设置（快捷键为：Crtl+Atl+s），之后选择Build、Execution、Deployment -> Build Tools -> Maven

其中需要将User setting file和Local repository设置为自己目录下的文件，建议将自己开发环境与公司开发环境的setting分开保存，按需设置。

IDEA下的Maven快捷操作

按照上述的方法配置完成后，我们便可以开始在IDEA上快乐的使用Maven了，说起IDEA的强大，相信老鸟们都能说上几嘴，但是在我实习的这段时间内，让我切实感到非常爽的技术IDEA下的Maven操作了，这里和大家分享一下。

如下图，IDAE中提供了一个Maven的边界操作选项，多数在编辑页面的最右侧。

其中最顶部的几个按钮是平常开发最为常用的，其分别是Reimport All Maven Project、generate Source and Update Folders For All Project、Execute Model Goal。在更改过项目中的pom文件后，可点击Reimport All Maven Project来刷新依赖文件。

其次是Maven项目下的Lifecycle，其中包含了大多数的Maven操作指令，我们只需要点击便可使用相对应的功能。如：clean、compile等。

Maven的坐标

节点 描述 举例
GroupId 工程组的标识。在组织和项目中是唯一的，通常是以组织/公司+项目组名构成 org.apache
ArtifactId 工程的标识。通常是工程的名称（GroupId和ArtifactId）在一起定位了依赖在仓库中的位置 tomcat
version 工程的版本号。 org.apache.tomcat: 1.0

上述的三个字段就类似于坐标的形式，可以在依赖仓库中准确的定位到所需的依赖。

依赖冲突

这里我介绍两种Maven中的以来冲突，其中一种是Maven可以自己解决的，另外一种是必须认为干预的。

相同版本间的依赖冲突

若项目中多个jar包同时引用了一个相同版本的jar包，则这时会产生相同版本的依赖冲突，但是这种情况Maven会帮助我们解决这种冲突问题。其主要用到了两种方法：

短路径优先

​ 本项目 -> A.jar -> B.jar -> C.jar

​ 本项目 -> X.jar -> C.jar

如果出现上述的情况，Maven会根据短路径优先的原则，对同时引用的C.jar，Maven会选择第二条路径下载引用依赖。

声明优先

​ 若引用的路径长度相同时，则谁在pom.xml中先被声明，谁就被使用。

上述的情况的前提都是在项目中引用了相同版本的jar包时，Maven才会自动采取对应的策略来进行处理，但是如果是引用了不同版本的jar包时，那Maven是不会自动解决的。

不同版本间的依赖冲突

当项目中引用了不同版本的依赖后所产生的冲突，这时就需要受到使用排除依赖来解决，具体操作见上一小节。

多模块项目
